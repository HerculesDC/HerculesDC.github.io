Dev Log

Space Shooter

April 6, 2021

Intro

This is a generic 2D top-down shooter/bullet hell project I'm prototyping using Python/pygame and P5.js on codepen.io.

The choice of languages/libraries stem basically from the fact that both are fairly easy to use. I use p5 to design the particles and other graphical elements I intended to use (apart from the spaceships). I then load the generated PNGs into pygame to see how they looked. In the future, my idea is to port the code to a C++ engine with SDL, and to use Lua as a scripting language.

pygame has many interesting features, one of them being the ability to save images from the application (via pygame's image submodule). This allows me to screenshot the game-making process.

All for the sake of learning...

A bit of storybuilding, as there are twelve ships, I originally decided to name them after things that came in twelves, like months of the year, nouns from 12-step programs, zodiac signs and the like. The problem arose when I noticed patterns in the ships, and my naming convention turned to mush. I then settled for Greek gods/titans names (sometimes with numbers), apart from what I intend to be the first two playable ships: Alpha and Omega. The names are listed in order in the table below:

Alpha Poseidon Gaia II Charon Hades
Artemis Omega Gaia V Selene
Gaia III Gaia I Gaia IV


These images were taken from Ubisoft's NEXT API from 2020. I'm using them as a model (and I have written helper programs that let me slice up spritesheets and save slice parameters in a .json file), but replacing these later will be a relatively trivial matter. This being a prototype, I simply want to flesh out a couple ideas before I port the code to C++, possibly with different graphical materials (don't trust my choice of art, though. I'm just a programmer).

The red rectangles represent each sprite's slice in the spritesheet. Since the last two ships are considerably larger than the others (which fit a 256*256 square), I ultimately decided to place slicing coordinates in a .json file. The blue rectangles (obtained using one of pygame's Surface methods) were later saved into a new .json file. This allows me to save sprite space by slicing and blitting only the smallest possible area (I don't intend to rotate the ships; at least, not in this first iteration). The green circle was outlined as more of a test as I brainstorm for more suitable collisions.

On a side note, the flames are separate sprites in the spritesheet. They were joined in-app.


This is a simple test of one of the shields I designed for the game. I call this one "moonshield", as it looks like a half-moon rotating around the ship's length. This shield is actually based on the Moon Phases project from a previous post. The numbers seen below each ship indicate the shield's opacity level, chosen from a range between 0 (no opacity) and 255(fully opaque). The shield is generated as a (32*256)*256 spritesheet, and the source code is available here.


This is a second shield model, which I (creatively) dubbed polyshield. I have not yet decided which one(s) to use, and I'll probably go back to the original code and create the frames directly in p5 before moving on with its implementation. I believe p5's rotation and image generation algorithms work somewhat better than pygame's rotate method.

You may also notice that there is a subtle difference between shields that have the same number of sides (I forgot to generate the second version for the hexashield). I wanted to have an "energy field" feel with the thicker lines and a "soap bubble" effect for the thinner ones. However debatable my level of success may be, and considering these effects were generated white on purpose (to allow for recolouring at runtime), this is enough for further testing. If you want to try it out, you can get the source code here.


These next two pictures show the game's particles as generated after being sliced into their respective sprites from their spritesheets. The first one shows the particles with colorkey; that is, a background colour that will be ignored when the particle is rendered on the screen. The second one shows particles with transparency. Transparency and colorkeys cannot be used in conjunction, so the second set had to be handled separately.
